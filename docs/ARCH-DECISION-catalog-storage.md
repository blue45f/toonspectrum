# 아키텍처 결정 — 카탈로그 저장/조회: **정적(CDN) 유지** (OCI 이전 후에도)

## v2 (2026-06): 카탈로그 **파일 전용** + DB는 **UGC 전용** — 서버 경로까지 DB 스냅샷 제거

### 배경(비용 근거: Neon 전송 쿼터 사고)
v1 이후에도 서버 ingest/폴백 경로는 24k편 카탈로그 JSON(~22MB 텍스트)을 `catalog_snapshot`
한 행으로 **통째로 쓰고 읽는** 구조였다. 크롤 1회당 쓰기 ~22MB + 폴백/검증 읽기 ~22MB가 누적되며
**Neon 무료 플랜의 데이터 전송 쿼터를 실제로 소진**시키는 사고가 났다(리뷰·로그인 등 동적 기능까지 영향).

### 결정
**카탈로그는 파일 전용, DB는 동적 데이터(리뷰·커뮤니티·계정·창작) 전용.**

- ingest(수동/주기/CLI)는 검증된 `Title[]`을 `apps/api/data/catalog.json.gz`(또는
  `WEBDEX_CATALOG_FILE`)에 **원자적으로 저장**(tmp 쓰기 → rename)하고 메모리 카탈로그를 즉시
  핫 리로드한다. 동일 runHash 면 파일을 다시 쓰지 않는다.
- API 부팅은 **gz 파일 → 없으면 빈 카탈로그**. 갱신 감지는 DB 폴링 대신 **파일 mtime/size
  스탯 폴링**(syscall 1번, 비용 0).
- `catalog_ingest_run` 실행 이력(상태·건수·runHash·오류, 행당 수 KB)만 DB에 남긴다 — 수동 크롤
  UI/status 가 소비.
- DB `catalog_snapshot` 읽기/쓰기/DDL 은 `WEBDEX_CATALOG_FORCE_DB=1` **레거시 플래그**에서만
  동작(롤백 가능성 보존). 잔존 대형 행은 `scripts/db-prune-snapshots.mjs` 로 정리한다.
- 부수 조치: pg 풀 슬림화(`max 3`·`idleTimeout 10s`·`allowExitOnIdle` — Neon autosuspend 유도),
  세션 사용자 조회 30초 마이크로캐시(요청당 SELECT 절감).

### 효과(전송량)
크롤 1회당 DB 전송 ~44MB(쓰기+읽기) → **~수 KB(실행 이력 1행)**. 부팅/핫리로드 시 DB 전송 0.


## 배경
Neon 무료 데이터 전송 쿼터 초과로, 카탈로그(웹툰·웹소설 ~30k)를 DB 조회 → **정적 스냅샷**
(`catalog.json.gz` → 빌드 시 `public/data/*.json` → Vercel CDN, 클라이언트 인메모리 검색/랭킹)으로
전환했다. OCI(자가호스트 Postgres, 전송 쿼터 없음)로 이전하면서 "DB 방식으로 되돌릴지" 재검토함.

## 결정
**정적(CDN) 방식을 유지한다.** OCI Postgres는 **동적 데이터 전용**(리뷰·커뮤니티·피드백·인증).
카탈로그를 DB로 되돌리지 않는다. (Oracle ADB 등으로의 전면 DB 리라이트도 하지 않음.)

## 근거 (비용·성능·갱신·복원력)
- **성능**: CDN 엣지 정적 파일 + 클라이언트 인메모리 쿼리 = 카탈로그 상호작용에 서버 왕복 0.
  DB 방식은 매 검색/랭킹/필터가 단일 OCI 서울 VM(1 OCPU)로 왕복 → 느리고 VM이 전 사이트 병목.
  (정적 기준 LCP ~265ms 측정됨.)
- **유지비용/부하**: 무거운 카탈로그 트래픽을 무료 CDN이 흡수. DB로 되돌리면 그 부하가 작은
  무료 VM 한 대에 집중돼 상향(=비용/운영 리스크) 유발. 정적은 VM을 가볍게 유지.
- **복원력**: 정적이면 OCI VM이 죽어도 **사이트 핵심(탐색·검색·랭킹)은 계속 동작**(리뷰/커뮤니티만 영향).
  DB로 되돌리면 단일 VM이 전 사이트의 SPOF가 된다.
- **데이터 갱신**: 정적도 충분히 신선하다 — OCI 서울 VM의 일일 크롤 cron → `catalog.json.gz` 커밋 →
  Vercel 재배포(~1분 전파). 카탈로그 메타데이터는 느리게 변하므로 일 단위로 충분. 갱신 주기의
  병목은 "저장 방식"이 아니라 "크롤 주기"라, DB로 바꿔도 더 신선해지지 않는다.
- **Neon 쿼터는 트리거였을 뿐, 정적은 그 자체로 더 나은 설계**였다. OCI로 옮긴다고 되돌릴 이유 없음.

## DB는 무엇에 쓰나 (동적 전용, 가벼움)
리뷰·커뮤니티·피드백·인증. 데이터가 작아 **self-host Postgres로 충분**(백업 pg_dump cron, 마이너
업그레이드만). 전면 관리형 전환(MySQL HeatWave 등)은 운영부담을 더 줄이지만 리라이트 비용이 있고,
이 정도 경량 데이터엔 과함. **Oracle ADB 전면 리라이트는 비권장**(최대 리라이트·락인, 이득 없음).

## DB 방식이 더 나아지는 조건(미래 트리거)
카탈로그가 100k+로 커져 클라이언트 페이로드(현재 ~382KB brotli)가 과해지거나, 서버사이드
페이지네이션/실시간 개인화 쿼리가 필요해지면 그때 DB-백엔드(서버 코드는 `VITE_CATALOG_SOURCE=api`로
이미 보존돼 토글 가능)로 전환을 재검토한다.

## 결과
- 카탈로그: 정적 유지(Vercel CDN). 크롤·검증·배포·알림 파이프라인은 OCI 서울 VM cron이 담당.
- 동적: OCI self-host Postgres. `/api/cover`·`/api/og`는 Vercel 엣지 유지(선택적 rewrite).
- 토글 경로 보존: 필요 시 `VITE_CATALOG_SOURCE=api`로 DB-백엔드 복귀 가능(되돌릴 수 있는 결정).
